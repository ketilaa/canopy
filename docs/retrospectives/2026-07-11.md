## Shipped

### Morning: `try-tools` proves `find_symbol`, then real wiring — see `2026-07-12.md` for the fuller account
- **`e0ca563`** — `try-tools` gained a second scenario, backed by a real Roots index (not a mock),
  reproducing a deliberately obscure missing-import shape (`conjureWidget`, not the easily-guessed
  `create<Entity>` pattern) in a generic scratch project. 4 of 5 real runs called the tool and
  produced the correct import — a real result, not a hunch, and the thing that actually justified
  building the production version. Extracted a shared `run_tool_loop()` so both `try-tools`
  scenarios stopped duplicating the request/response cycle.
- **`1d3f372`** — `find_symbol` wired into the real fix loop (`fix_file_with_tools`, a bounded
  4-iteration loop), gated to TS/TSX files with a parsed `TS2304` symbol name specifically. Two
  things fixed before calling this safe to rely on: each tool call got its own nested console
  line (a silent tool call would have been a step backward from the existing fix-loop checklist),
  and `llm-debug.log` started logging tool-call requests/responses unconditionally — it hadn't
  been logging them at all until this point.
- **`87819ee`** — cut real prompt noise the tool now made obsolete (duplicate import-path
  arithmetic, an `isolatedModules` rule, a necessarily-incomplete static package blocklist replaced
  by a real fact read from `package.json`) — and, found mid-flight rather than sought out, a
  genuine architectural correction: `canopy-cli` was linking `roots-storage`/`roots-context`
  directly, coupling it to Roots' internal schema with no interface boundary. Rewritten to shell
  out to the `roots` CLI, the same pattern `init`/`index` already used, with one documented,
  deliberate exception (`find_test_call_shape`, which parses in-memory content before anything is
  ever indexed). CLAUDE.md's escalation-order principle (tool/fact → prompt → gated code, never
  skip a rung) was written down this same day, as the lesson of the day it happened.
- **`36061a3`** — a real, live-observed Green-phase regression: a correctly-grounded single-
  parameter stub got reverted to five positional parameters by Green phase, because Green phase had
  zero visibility into its own stub (`current_content` hardcoded to `None`, `observed_call` not
  threaded through at all). Fixed by giving Green phase the same arity-grounding fact stub
  generation already had.

### Evening: tool access reaches test-gen/stub-gen, closing a real dogfooding bug — not covered elsewhere
- **`eb27573`** — added a second tool, `read_file`, sandboxed to the service directory, offered
  unconditionally alongside `find_symbol` rather than gated behind detecting a specific error class
  first — "the model choosing not to call a tool it doesn't need costs nothing." Stated an explicit
  tool preference directly in both tools' own descriptions (not in skill prose elsewhere) to guard
  against the model defaulting to the "safer" full read even when it doesn't need one.
  `canopy-prompt-reviewer` caught a real house-style violation in the first draft — the `read_file`
  example used domain-specific "Product" vocabulary where the generic-placeholder rule requires
  "Widget."
- **`e2687a6`** — closed the day by digging into a real dogfooding failure: `Product.test.ts` mixed
  calling conventions, correct positional calls in the happy-path tests but an incorrect single-
  object-argument call in the "missing mandatory field" tests. The rule that should have prevented
  this already existed, correctly worded — but lived 15 bullets deep in a generic list at the end
  of an already-long prompt. The first fix attempt added a worked example to dead code for this
  stack specifically (caught by `canopy-prompt-reviewer`); the real fix landed a third worked-
  example test directly next to the two Node/Express model-layer tests it needed to stay
  consistent with. Separately, in the same commit: "ALWAYS call find_symbol" pointer text was
  reaching prompts (test-gen, stub-gen) with zero tool-calling support at all — pure noise,
  replaced with a shared `tools_hint_section()` that only renders when the specific call's `tools`
  slice actually contains that tool. Test-gen and stub-gen also gained real `find_symbol`/
  `read_file` access in this same commit, since by stub-gen time the sibling/test files already
  exist on disk and a real lookup beats inferring shapes from the entity schema alone.

## Learned

**A tool proving itself in a throwaway experiment first, then getting wired into production, held
up as the right order again.** `try-tools`' second scenario (`e0ca563`) is what actually justified
`find_symbol`'s real wiring a few hours later — not a hunch about what "should" help.

**An architectural coupling can be found while fixing something unrelated, not just when looking
for it.** The Roots-as-library correction (`87819ee`) surfaced mid-flight during a prompt-noise
cleanup pass, not from a dedicated architecture review.

**Proximity matters as much as correctness.** The rule that should have prevented the
`Product.test.ts` bug (`e2687a6`) was already correctly worded — it was just 15 bullets away from
the code it needed to govern. A correct rule in the wrong place is functionally the same as a
missing one.

**A tool offered somewhere it structurally can't work is pure noise, not a defensive nicety.**
Twice this day (the "ALWAYS call find_symbol" pointer text reaching tool-less prompts) the fix was
simply to stop sending an instruction the model had no way to act on, gated by whether the tool was
actually present in that call's own `tools` slice.

**`canopy-prompt-reviewer` earned its keep twice in one day**, catching a dead-code worked example
(a fix that could never render for the stack it targeted) and a generic-placeholder violation — two
different classes of defect, both real, both in prompt-facing changes that would otherwise have
shipped silently wrong.
