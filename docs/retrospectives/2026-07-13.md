## Context

The day opened with two cleanup threads left over from the prior audit work (architecture-skill
threading, a rename freeing "contract" as a term), then pivoted into the day's real subject: the
behavior-first planning redesign — Story → Behaviors → Decisions → Clusters → Contracts — went
from a design doc to a fully implemented, five-stage pipeline (Stage 0 through Stage 4), live-tested
against a real dogfooding project the same day it was built. That led straight into a first

scenario generation away from schema/policy generation — not yet finished.

## Shipped

### Renamed the OAS artifact to free "Contract" for the new pipeline
Pure rename: `contract.yaml`/`contract_yaml`/`generate_story_contract` (the OpenAPI/HTTP artifact)
became `openapi.yaml`/`openapi_yaml`/`generate_story_openapi` across storage, LLM, and CLI layers.
No behavior change — just clearing the word "Contract" so the new pipeline's own Stage 4 artifact
could use it without collision.

### Implemented the full behavior-first pipeline: Stage 0 through Stage 4
Five new REPL commands, each with its own types, storage, prompts, and gating logic:

- **Stage 0 (Specification Completeness)** — checks a story's entity schema, scenarios, and open
  questions for gaps (`missing_scenario`, `ambiguous_outcome`, `unresolved_question`), gates on any
  blocking gap. First version asked the model to compare schema and scenarios holistically in one
  pass; against a live spec with 9 known constraint-coverage requirements it found 4, missing the
  same constraint shape on one field after correctly catching it on three others. Rebuilt around
  three explicit, mechanically-enumerated checklists — one entry per field-constraint pair, one per
  scenario, one per open question — and the re-run found all 9. Also fixed the same day: gap
  severity (blocking vs. advisory) is now computed deterministically from the gap's own category
  rather than asked of the model, removing one more degree of freedom it could get wrong.
- **Stage 1 (Behavior Extraction)** — turns a spec into atomic, taggable behaviors (mechanical
  validation/construction/event behaviors derived directly from the schema, plus scenario-inferred
  integration behaviors), with a coverage audit confirming every scenario produced at least one
  surviving behavior.
- **Stage 2 (Decision Extraction and Gating)** — elevates unresolved business questions to a
  first-class, gated `DecisionPoint` artifact rather than a note a later stage could silently work
  around. Contract generation may not proceed for behaviors blocked by an unresolved decision.
- **Stage 3 (Mechanical Clustering)** — groups behaviors into unit clusters (one per subject/kind
  pair) and one integration grouping, with an LLM review pass over the mechanical baseline that
  caught, among other things, a vacuous constraint (`min_length: 0`) that had produced a
  self-contradictory behavior statement ("shorter than 0 characters is rejected").
- **Stage 4 (Contract Generation)** — one contract per cluster/grouping, with a dependency-review
  pass that reliably caught dependencies the mechanical substring-matching baseline missed (a
  workflow's error-translation text says "Name," not the compound "ManufacturerName" a validation
  cluster is keyed on) — confirmed as a repeatable gap, not a one-off, against two separate stories.

Both crash bugs found and fixed along the way: `generate_stories_from_intent` on a fenced LLM
response, and `identify_architectural_questions` on an explicit `null` in a list field (previously
failed the whole batch atomically; now parses proposals item-by-item so one malformed entry doesn't
lose five good ones).

### First real dogfooding run against the new pipeline
Ran `intent` → `spec` → `behaviors` end to end for a small, deliberately new story — registering a
manufacturer — driving every interactive gate through a scripted pseudo-terminal session. Found and
recorded, in a structured audit report, six real findings: domain-vocabulary extraction has no
fallback when a story's `want` field doesn't name the entity (it invented `User`/`UserRegistered`
from an under-specified `want`); the same architecture and story shape produced a different ADR set
between two separate sessions with a visible downstream effect (no domain-event ADR meant zero
event-shape behaviors); Stage 0's own constraint checklist repeated identical false gaps against
scenarios that directly addressed them, in the same response where it correctly echoed those same
scenarios back for a different checklist; Stage 0's gap-finding wasn't deterministic run-to-run
against an unchanged spec; a vacuous constraint produced a nonsensical behavior statement, caught by
Stage 3's own review; the mechanical dependency-inference baseline reliably missed validation
contracts specifically, for a diagnosable structural reason.

### A first reproducibility sweep, and the most severe finding of the whole redesign so far
Ran the same intent statement through `intent → spec → behaviors` three times from a clean slate,
with identical (default) gate answers throughout, to isolate LLM sampling variance from human
curation. Two runs correctly derived "manufacturer" as the story's domain; one derived "account."
All three runs echoed the literal phrase "register an account" from a rule's own worked example,
regardless of what the story was actually about — a systematic anchoring failure, not a fluke. In
the one run that also picked "account" as its story ID, the anchoring went further: the model's
generated data schema for what should have been a Manufacturer entity — with "Manufacturer" named
correctly in both the story and the domain vocabulary shown in the same prompt — came back as a
full user-account schema (username, password, email). Nothing else in that run's input pointed
toward an account of any kind.

### Fixed the anchoring rule and added Entity Continuity
Rewrote the `want`-naming rule to require consistency with the actual intent statement, blacklisted
generic filler nouns ("account," "record," "item"), and replaced the single fixed example with a
conditional pattern spanning several different hypothetical intents, so there's no one
decontextualized phrase to copy. Live-verified: the same intent that previously produced "register
an account" now produces "register a manufacturer." Added `check_entity_continuity` — a mechanical,
non-LLM gate run immediately after spec generation: does the entity the model just generated match
an entity already established in the project's own domain vocabulary? If vocabulary already exists
and nothing matches, the operation fails loudly and nothing is saved. Covered by 5 unit tests
including the exact Account-vs-Manufacturer scenario from the sweep.

### Added Event Continuity, the sibling check
Same shape, same placement, one stage later: does a newly-accepted domain-event ADR's event name
share the same entity prefix as an entity already known to the project? Deliberately does *not*
gate on the exact verb (Created vs. Registered for the same entity), since the sweep showed that
wording varies legitimately per the domain-extraction rules' own ambiguity — gating on it would
just produce noise. 5 unit tests, including the exact `AccountRegistered`-vs-`Manufacturer`
scenario and a healthy non-gating case.

### Started splitting scenario generation away from schema/policy generation
End-of-day work, not yet fully landed: `story_spec_prompt`'s single holistic call (entity schema +
business-policy checklist + scenario writing, all in one response) was split into two calls, with a
mechanically-computed coverage matrix in between driving the second call's scenario writing — the
same "enumerate, don't review holistically" fix already proven at Stage 0, applied one level
earlier in the pipeline. Also formalized, in the project's own house style, the general rule this
keeps confirming: prefer exhaustive enumeration over holistic review for any bounded set, and
distinguish a deterministic audit (compare against a known fact, fail loudly — Entity/Event
Continuity's own shape) from Rust-side compensation (silently rewriting model output, which stays
forbidden).

## Learned

**A prompt's worked example is a candidate answer, not a neutral illustration — and this was worse
than expected.** We'd already picked a "safe," generic-sounding example noun ("register an
account"), not a domain-specific one, expecting that to be enough. Three separate runs all echoed
it verbatim regardless of the actual story, and one derailed an entire generated schema into an
unrelated domain. The fix wasn't a better noun. It was removing the single fixed example entirely
in favor of a pattern shown across several different hypothetical subjects at once.

**A model can hold the correct answer in its own context and still drift away from it.** The
Account-schema run had "Manufacturer" stated correctly in two separate places in the same prompt —
the story and the domain vocabulary — and produced an entity schema for something else entirely.
Showing the model correct context is necessary. It is not sufficient. That's the whole argument for
a mechanical, post-hoc check rather than trusting a better prompt alone.

**Holistic review keeps failing the same way, and enumeration keeps fixing it the same way.** This
is now confirmed at Stage 0 (4/9 → 9/9 constraint gaps) and it's the explicit reason the day ended
mid-way through applying the identical fix to scenario generation itself. Worth treating as a
default lens for any future prompt reliability work, not re-discovering per prompt.

**Independent reproducibility sweeps turn "this feels flaky" into a number.** The account/
manufacturer divergence wasn't reported because someone eyeballed a bad run — it showed up because
we deliberately ran the same input three times and diffed the outputs. That discipline is worth
keeping as new pipeline stages land, not just applying once at the end.
