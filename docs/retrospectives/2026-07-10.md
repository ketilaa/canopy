## Shipped

### The day started by retiring its own earlier automation
The first six commits (`wip: auto-install checkpoint`, 08:05–09:14) were produced by an earlier,
more automated version of the checkpoint-commit mechanism — auto-stage, invoke a nested `claude
-p` to draft and gate the message, commit, reinstall, all without a human in the loop at session
end. Claude Code's own safety classifier rejected that shape as a self-modification risk
(unsupervised commit-and-deploy, decided by a background agent nobody was watching). Replaced it
with the design that shipped today: a Stop hook that only *reminds* — no git command, no install,
no nested agent call, hash-gated so it fires once per distinct diff and can never block a session
from ending. Judgment about whether to commit stays with whichever session is active; the hook
just makes sure that judgment gets exercised instead of silently skipped. Documented the full
arc (why the automated version existed, why it was rejected, what replaced it) in CLAUDE.md so
the reasoning survives past today.

A related bug surfaced immediately: the reinstall hook checked *git dirty-state* to decide
whether to rebuild, but the new commit discipline means the tree is usually already clean by the
time Stop fires — so real prompt changes went uninstalled. Fixed by content-hashing the actual
`.rs`/`Cargo.toml` files against a marker file instead of asking git, making it agnostic to
whether the change happened to already be committed.

### A run of real skill/harness bugs, found from live evidence
- **Repository skill's worked example primed speculative CRUD** the model never needed —
  trimmed to only what the story actually required.
- **Stub-over-implementation waste**: the Red phase was treating a stub that came back fully
  implemented (ignoring the "stub only" instruction) as just another compile pass, then racing
  into Green on top of it. Added `RedSanityOutcome` (`ExpectedRed` / `AlreadyImplemented` /
  `Broken`) so an accidentally-correct stub short-circuits straight past Green instead of risking
  a regeneration that could break what was already right — confirmed later this really happened:
  an `exactOptionalPropertyTypes` violation got introduced into an otherwise-correct file purely
  because Green regenerated it unconditionally.
- **Stub/test signature-arity mismatches**, twice — including a regression after the first
  prompt-only fix. This is the one that eventually got a *deterministic* fix instead of a third
  round of prompt escalation (see below).
- **Green phase had no compile-check gate**, only a Jest run — added one, symmetric with Red.
- **Validation scenarios were being generated for the event layer**, which has nothing to
  validate — skipped at the source instead of teaching the model to recognize the mismatch.
- Console UX kept paying for itself again today: floating `println!` diagnostics for fix-loop
  narration, closing messages, and self-report notes all got folded into their owning step's
  tree entry instead of scrolling past disconnected from context.

### The honest conversation: is prompt engineering hitting a ceiling?
After the fourth or fifth instance of the same *class* of non-compliance (stub arity, scenario
skipping, `exactOptionalPropertyTypes`) surviving a well-worded, correctly-positioned prompt fix,
the real question became explicit: are tools like Claude Code actually driving implementation
this deep in the weeds, or is Canopy asking a 14B local model to do something structurally
unsuited to it? Worked through several bigger ideas without committing to any of them yet — an
implementation-review agent, flipping Green-before-Red, a shared step-plan divided across test
and implementation prompts — and landed on the smaller, sharper question underneath all of them:
**can Roots deterministically ground test/implementation agreement instead of asking the model to
self-check?** That reframing is what unlocked the rest of the day.

### Roots grounds stub signatures deterministically
Added `find_subject_calls`/`ObservedCall` to `roots-parser` (tree-sitter, verified field names
against the actual grammar's `node-types.json`) to extract exactly how a test calls the subject
under construction. `canopy-cli` collapses this to one fact only when every call site agrees on
method + arity, and hands it to the stub-generation prompt as a stated fact rather than an
instruction to re-derive. This is the deliberate shift the day's ceiling conversation pointed
at: stop asking the model to count arguments correctly on every call, compute the true answer
once in Rust and hand it over. Confirmed working in a real `implement` run afterward.

### An experimental tool-calling capability test, and a fallback parser born from it
Before building anything bigger on the assumption that giving the model tool access would help,
built `canopy try-tools` — a minimal, clearly-experimental REPL command with one tool
(`read_file`, sandboxed) and a scratch file with an unguessable function signature the model
can't pattern-match from training data. A correct final answer is real evidence of tool use, not
a lucky guess.

First real run: the model got the tool name and arguments right every time, but never emitted the
exact `<tool_call>` tag its own loaded chat template asked for — it echoed the sibling `<tools>`
tag used to *present* tools to it instead, or dropped the wrapper entirely. Built a fallback
parser for the plain-text content the structured `tool_calls` field left empty. Five follow-up
runs against that first version: 4/5 recognized correctly, one new near-miss shape (the model
echoed its own tool *spec* — no `arguments` key, correctly rejected — then appended the real call
as a second, unwrapped JSON object later in the same response, which the single-candidate parser
never looked at). Generalized the parser from "one wrapper tag or the whole string" to a
string-literal-aware brace scanner that extracts every balanced top-level JSON object in the
content and tries each in turn — subsuming the tag-based logic instead of adding a third special
case, and handling whatever the model's formatting drifts to next without another hardcoded
variant.

## Learned

**A ceiling conversation is more useful when it ends in a falsifiable test, not a decision.**
"Is this model good enough" had no clean answer in the abstract — five philosophical framings
were floated (review agent, reordered TDD, shared plan) and none was obviously right. The
question that actually moved things forward was narrower and answerable: can Roots ground one
specific fact deterministically? Yes. That's a pattern worth repeating the next time a "maybe
we're approaching this wrong" feeling shows up — look for the smallest falsifiable sub-question
before picking a big redesign.

**Deterministic-fact-injection beats a third round of the same prompt fix.** Two prompt-only
attempts at the stub-arity bug (one of them regressing) taught the same lesson `exactOptional
PropertyTypes` taught yesterday from a different angle: if a fact is mechanically computable,
computing it in Rust and stating it is more reliable than asking the model to re-derive it
correctly every single time, no matter how well-worded the self-check instruction is.

**A capability gap and a formatting gap look identical from the outside — and needed a
purpose-built probe to tell apart.** Five straight `try-tools` runs got the tool name and
arguments exactly right; only the wrapper tag was ever wrong. Without an isolated,
training-data-proof probe, this would have been indistinguishable from "the model can't reliably
use tools at all" — a much bigger and wronger conclusion, and one that could have justified
abandoning the tool-access idea entirely instead of just handling its output more tolerantly.

**A tolerant parser should generalize the shape of the leniency, not enumerate instances of it.**
The first fallback fix hardcoded the two tag names actually observed. The very next batch of runs
produced a third shape neither tag anticipated. The fix wasn't a third hardcoded case — it was
noticing that all three observed shapes are instances of one more general fact (a JSON object
appears somewhere in the text) and parsing for that instead.

**Retiring your own automation is itself worth documenting, not just doing quietly.** The
auto-commit mechanism this session replaced wasn't a mistake to bury — writing down why it
existed, what specifically made it unsafe (unsupervised commit-and-deploy, decided by a
background agent nobody was watching), and what replaced it means a future session won't
accidentally reintroduce the same shape while trying to make checkpoint commits more convenient.
