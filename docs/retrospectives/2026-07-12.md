## Shipped

### find_symbol and read_file wired into the real pipeline
Landed the `find_symbol` tool in the fix loop for real (not just the `try-tools` experiment),
enhanced its answer to include a computed relative import specifier and a type-only flag so the
model never has to count `../` levels or reason about `isolatedModules` itself. Added a second
tool, `read_file`, so a fix attempt can pull any file's content on demand instead of every
plausibly-relevant file being pushed into the prompt whether or not that attempt needs it — this
replaced a raw full-file-content fallback that used to fire whenever Roots' compact surface
wasn't available. Both tools now reach test-gen and stub-gen too, not just the fix loop, since by
stub-gen time the test and sibling files already exist and a real lookup beats inferring shapes
from the entity schema alone. Deleted the now-redundant `try-tools` experimental command — it did
its job.

### A real architectural correction: canopy stopped linking Roots as a library
Mid-session, direct user correction: "Do not mix roots into canopy. canopy only calls roots as
an external command." Traced the actual scope — `canopy-cli` was linking `roots-storage`/
`roots-context` directly (`Store::open`, `query_exact`, `feature_context`), coupling canopy to
Roots' internal SQLite schema with no interface boundary. Rewrote `roots.rs` to shell out to the
`roots` CLI (`symbol`, `dump`, `context --feature`) and parse its JSON — the same pattern
`ensure_indexed`/`reindex` already used for init/index — and removed `roots-storage`/
`roots-context` from `canopy-cli`'s `Cargo.toml` entirely. One deliberate, documented exception:
`find_test_call_shape` still links `roots-parser` directly, since it parses a test file's
in-memory content before it's ever written to disk or indexed — there's no live index or CLI
command involved, just a parsing utility. Verified end-to-end against the real dogfooding project
before considering it done: `module-surface (roots index)` context and the `find_symbol` tool
both worked correctly through the new shell-out mechanism.

### Diagnosed and fixed a real Green-phase regression
A service method's Red-phase stub correctly declared a single object parameter (grounded by
yesterday's `observed_call` fact), but the Green-phase implementation reverted it to five
positional parameters — one per entity-schema field — even though the test still called it with
one argument. Root-caused by actually reading the call site, not guessing: `execute_implementation
_with_test` was hardcoding `current_content: None` and never received `observed_call` at all, so
Green phase generated "from scratch" every time with zero visibility into the stub it was
replacing. Its only instruction was "use the exact signatures shown in the sibling section" —
which never contained the file's own signature, since a file isn't its own sibling. Fixed by
threading `observed_call` into Green-phase generation and re-reading the actual (possibly
fix-loop-corrected) stub content from disk instead of passing `None`.

### The big one: a three-agent parallel audit of the entire prompt/skill system
The user's challenge — "look with different eyes... Canopy is guiding at a too detailed level" —
came with a sharp concrete example: a single dense sentence ("implement a pure model... factory
for valid instances only") conveys more real constraint than a pile of itemized rules addressing
symptoms one at a time, because it invokes a concept the model already has strong, correct priors
about from training. Rather than take that on faith, ran three parallel research agents (one per
file: `tech_stack.rs`, `testing.rs`, `step.rs`+`fix.rs`) to independently catalogue every rule as
concept-derivable / arbitrary-and-necessary / redundant-in-place / unclear. All three converged,
unprompted, on the same missing piece: nowhere did canopy state that a model factory is a pure
function with zero dependency on other layers — instead there were ~6 scattered symptom-rules
(no `new`, no `.save()`, positional-not-object args) that are each a *consequence* of purity,
never the cause. That gap is exactly what let the day's reported bug through (a model-layer test
mocking `ProductRepository`/`EventPublisher`).

Landed in priority order:
- **3 outright bugs**, unrelated to noise but just wrong: a shared "always named exports" rule
  contradicting the Route layer's own required default export; a stale rule about `EventPublisher`
  ownership filed under `layer_rules["middleware"]` (never reaches a route file, per how
  `render_for_layer` scopes rules) that directly contradicted the Route layer's own correct rule;
  a worked example citing "the Route rules above" — text that doesn't exist anywhere in the
  render chain.
- **~10 duplicate removals**, each verified to *literally always* co-occur in the same rendered
  prompt, not just "similar": "no code fences" stated three ways on top of the shared output
  contract already saying it, jakarta/javax repeated between two files, "mock only the service
  layer" repeated between two files, a stub rule stated three times in a row. Two auditor-flagged
  "duplicates" were deliberately left alone after checking they were the *only* copy reaching
  React/Angular test-gen — a case where blind mechanical removal would have silently broken a
  different tech stack.
- **The core fix**: `tech_stack.rs`'s Models section now leads with "A model is a PURE factory...
  NEVER import, reference, or depend on any other layer... no I/O and no state beyond the object
  it returns," reinforced in `testing.rs`'s model test rules with the test-side consequence
  ("nothing to mock — never `jest.mock()`, never reference a repository/service/event publisher").
- **A schema/contract layer-gating fix**, found while answering a direct, sharp side-question
  ("only the model needs spec.yaml, only routes need contract.yaml, right?"): test-gen was
  sending the full `entity_schema` to *every* layer unconditionally — zero gating at all, unlike
  stub/Green-phase which already excluded some layers. Confirmed the OAS contract-generation
  prompt already maps `entity_schema`'s own validation constraints onto the contract schema, so
  routes getting both was genuinely redundant. Tightened both prompt paths to send `entity_schema`
  only to the model layer, `contract.yaml` only to route/api-client.

`canopy-prompt-reviewer` caught one real regression from the schema-gating change on its own:
two rules (one in each of two files) still told the model to "ignore the Entity schema below" for
the event layer — text that, after the gating fix, is never sent to the event layer at all.
Reworded both to state the underlying rule without presupposing a schema section that no longer
existed. Exactly the kind of subtle, easy-to-miss regression a second pass of review is for.

### Live validation: real, measurable improvement, plus one narrower residual bug
Reset the whole plan and ran `implement` fresh against the audited prompts. Red phase — test-gen,
stub-gen, compile-check, sanity-check — passed with **zero fix-loop iterations**, the first time
all session a step got through Red this cleanly. No purity violation, no arity mismatch. A
narrower, different bug surfaced in Green phase instead: `categories` carries two independent
schema constraints (`max_length` per item, `max_items` for the array), and the model wrote a
correct `max_items`-boundary test but asserted the *other* constraint's message text. Checked the
actual prompt: the disambiguation rule ("never borrow the other constraint's number or message
text") is present and correctly worded — a genuine compliance gap on an existing, well-placed
instruction, not a missing one. Logged to the backlog rather than chased immediately, given the
user was heading to sleep.

## Learned

**Independent convergence across three isolated readers is strong evidence, not a coincidence
worth ignoring.** None of the three audit agents could see each other's work. All three
independently flagged the same missing "pure factory" concept from three different files. That's
a meaningfully stronger signal than one person's read of one file — worth remembering as a pattern
for validating a hypothesis about a *system*, not just a single rule.

**A duplicate finding needs a "does this rule reach anywhere else" check before deletion, not
just "does this look repeated here."** Two rules the audit flagged as redundant ("jest is a
global," "plain string literal test IDs") turned out to be the *only* copy reaching React/Angular
test-gen — the apparent duplicate was actually load-bearing for a different tech stack the
immediate bug report never touched. Mechanically deleting "obvious" duplicates without checking
their full reach would have silently broken something else. Checked with a plain grep before
touching either.

**Removing noise can regress correctness in a way that's easy to miss precisely because the
removal is correct in isolation.** Gating `entity_schema` away from the event layer was the right
call — but two *other* rules were written assuming that schema would always be present ("ignore
the Entity schema below"), and deleting the schema section made those rules refer to something
that no longer exists. `canopy-prompt-reviewer` catching this is exactly why the review step isn't
optional for prompt changes, even ones that feel like pure subtraction.

**A live run that fails on a narrower, more specific bug than before is progress, not a wash.**
Today's validation run still broke — but on a single message-text consistency slip in a genuinely
tricky dual-constraint field, after a run that got all the way through Red phase clean for the
first time. The right response to "it still failed" is to check *what kind* of failure, not just
whether one occurred.

**A user's sharp example is worth testing against the actual codebase before agreeing or
disagreeing with the general claim.** "A dense concept-sentence beats a pile of itemized rules"
could have stayed a nice-sounding opinion. Auditing the actual prompt text and finding the model's
own worked examples never stated the "pure factory" concept anywhere — that's what turned an
opinion into a concrete, shippable fix.

  check can run against it, and it raises a real question about who's authoritative for

  of the gap before reaching for it.
- **The domain-object design tangent** (Product as interface+factory vs. a class with
  `fromJson()`/`toModel()`): concluded the anti-corruption-layer instinct is sound and currently
  unmechanized in the generated code (handled ad hoc at the route/service response-shaping layer),
  worth a `toProductView()`-style free function at some point — but not urgent, not started.
- Whether the "pure factory" framing pattern generalizes to *other* layers (service, repository)
  as cleanly as it did for the model layer is untested — today's fix and validation were scoped
  to the model layer specifically, since that's where the actual reported bug lived.
