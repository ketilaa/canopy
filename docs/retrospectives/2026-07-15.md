# 2026-07-15 — What we achieved and learned today

A working log, not a polished piece — raw material for later storytelling, kept honest about
what actually happened rather than smoothed into a narrative.

Cross-cutting, project-level-milestone day — the first entry written under the trigger-based
Knowledge Capture Cadence itself, and about a day that crossed one of its own trigger conditions
(first production wiring), not a story-scoped sweep (those stay in `docs/reports/`).

---

## Context

The whole day was one long, disciplined escalation: from "is the current contract schema
sufficient to drive implementation" through four numbered experimental stages to "contract-driven
file discovery now ships inside `canopy implement`." Every stage before the last ran as a
standalone, throwaway-safe cargo example that could never touch production code even if its
result was wrong. Only the fourth stage, after three stages of accumulating real evidence, touched
`canopy-cli` at all — and even then, behind a mechanical trigger with a provably-unchanged
fallback for every story that doesn't have it. The day closed with a second, different kind of
task: formalizing the knowledge-capture process itself into a trigger-based model, prompted
directly by noticing how unevenly the four capture-artifact types actually got used across this
same investigation.

## Shipped

### Contract Readiness Assessment, then Option 2's grounding fields
Hand-traced a real contract through the actual mechanical Rust functions and found three concrete
gaps — no file target, no canonical field identifier (a compound entity/field name is genuinely
ambiguous from a PascalCase-concatenated string alone), no mandatory/optional signal. Chose the
narrowest of three schema options considered: split `Contract`/`Behavior`'s existing `subject`
into explicit `entity`/`member` fields at the exact point they were already known separately,
never by re-parsing the model's own output. Added `kind: Option<BehaviorKind>` the same way.

### A live, reproducibility-tested probe settled the third gap empirically
Rather than assume mandatory/optional needed its own field, ran the same minimal contract through
the local model three times with nothing but its own `required_tests` prose. Three different
outcomes — including an under-constraining miss on the *mandatory* field nobody predicted going
in. That result, not intuition, justified adding `mandatory: Option<bool>` — the same mechanical
shape as `entity`/`member`, populated where `is_mandatory` was already a known local variable.

### Stage 1: a single contract, given to a model in isolation, wasn't enough
Real, freshly-generated `contracts.yaml` for `manufacturer-001` (required completing a spec gap
the story's own scenarios never covered — boundary conditions). Gave a model one validation
contract, an explicit "don't invent unrelated fields" instruction, and asked it to write the file.
2 of 3 runs added unauthorized JPA persistence annotations (`@Entity`/`@Id`/`@GeneratedValue`)
anyway, despite the instruction being present and correctly worded.

### Stage 2: full-file contract visibility eliminated it — the day's central finding
The real `manufacturer-001` contracts already had six contracts sharing one target file. Showing
a model all six at once, instead of one, produced 3 of 3 clean runs — the exact fields six
contracts authorize, nothing more. A second, unrelated failure mode (Bean Validation annotations
with no triggering mechanism) persisted regardless, exactly as predicted before running the
experiment — confirming the fix was specific to ownership, not a general prompt improvement.

### Stage 3: real compilation, not eyeballed LLM output
Wrote the generated files into a real, standalone Maven project and ran `mvn clean test` for
real. Caught two real bugs along the way, neither in the contract model: the harness's own naive
code-fence extraction corrupted a test file that had leading prose before its fence, and the
Spring Boot skill's new fix (added the same day) was keyed under a layer string neither real
generation call site actually produces — `detect_layer()` had no recognition of any JVM package
convention at all, a pre-existing gap this was the first thing to make visible. Fixed at the root
(`detect_layer()` itself), not worked around. 2 of 3 runs: real, compiled, executed,
zero-iteration passes.

### Stage 4: contract-driven discovery landed in production
`generate_story_plan_from_contracts` (new, mechanical, zero LLM calls) now runs automatically
whenever a story's `contracts.yaml` exists — a mechanical fact, not a flag — with a printed
message either way and `--legacy-planner` to force the old path. An independent safety review
before shipping caught and got fixed two real bugs (a JVM package dot-vs-slash path error; silent
dropping of integration-scope contracts instead of triggering fallback) — neither had fired
against real data yet, both were caught by review rather than a live failure.

### Contract Composition Assessment: finding the next frontier
Grounded directly in the current repository (confirmed by grep, not memory) that Stage 4 wired
*discovery* only — every real file's *content* is still generated by the full story/spec/
scenario/ADR prompts regardless of whether contracts exist. Named this as the single most
important open question, ahead of composition, multi-service support, or any schema question.

### Formalized the knowledge-capture cadence itself, from trigger, not calendar
Noticed directly from this same day's own work that the four (now five, counting `reports/`)
capture-artifact types fire at genuinely different rates, and that the existing CLAUDE.md
language already treated narratives as a trigger-based exception to an otherwise daily-anchored
habit. Generalized that exception to all five types uniformly, with per-type trigger conditions,
replacing the daily-checkpoint framing entirely.

## Learned

**A schema gap and a visibility gap can look identical from one contract's point of view.**
Stage 1's failure (unauthorized field invention) and Stage 2's fix (show every contract sharing
the file) are the clearest instance yet of a distinction this project's house style already
names but hadn't tested this sharply before: the model wasn't missing a fact about *that*
contract — it was missing *other* contracts that would have told it where its own scope ended.

**Confirming evidence and new evidence are different events, and only one changes a confidence
rating.** Stages 3 and 4 both reconfirmed Stage 2's finding — real compilation, then production
wiring — but neither added a second story or entity. The
`implementation-ownership-requires-full-file-scope-visibility` principle's confidence correctly
stays `medium`: more evidence of the same shape doesn't earn `high` the way genuinely independent
evidence would.

**Reviewing before shipping caught real bugs on both production-facing changes today, not
stylistic ones.** The Spring Boot skill's layer-key mismatch and Stage 4's dot-vs-slash path bug
were both real, silent-until-triggered defects a first pass missed and a second, adversarial pass
caught — not hypothetical thoroughness, concrete bugs that would have shipped otherwise.

**A knowledge-capture policy is itself subject to the same "trigger, not calendar" lesson it's
trying to state.** The clearest evidence for today's CLAUDE.md change was noticing, in the
middle of the exact investigation this retrospective describes, that a principle should have
been updated with new evidence and wasn't, until the trigger model made that gap visible.

### Learned (from the rest of the day)

**A false prediction caught before running anything is still evidence, not a wasted step.** The
Composition Assessment's ADR-fix prediction was checked against the actual mechanical rules first
and found wrong — cheaper to learn that from reading code than from a failed live run, and the
correction was recorded openly rather than quietly revised.

**Writing a regression test surfaced bugs a hand-verification would have missed.** Two of the four
composition-blocker fixes (`event_name` always `None`; the dependency rule storing the wrong id
namespace) were only found because a test was written for the first two fixes, not because they
were being looked for.

**A sharper, more specific sweep result changes a roadmap more than a vaguer one would.** The
sweep didn't just confirm "recommendations vary" — it named domain-event proposals as the single
least-stable output, which is what let the roadmap update promote the Human-Insight Inventory with
a concrete starting point instead of a vague next-in-line justification.

**An "evaluate this hypothesis" task can conclude "too early to tell" and still be a complete,
useful piece of work** — not a stalling non-answer. Both the domain-boundary hypothesis and the
Decision Point comparison actively ruled things out (a naming-convention connection is real; two
of three domain-event sub-decisions aren't fabrication-shaped) even though neither reached a final
yes/no. Precision about *what's* still unknown is itself the deliverable when the evidence is
genuinely mixed.

**Live verification caught a bug unit tests were structurally unable to find.** The classifier's
unit tests were built from the one real project's exact title wording; a second, independently-
generated run immediately exposed a blind spot that shape of test can't see by construction — not
a reason to distrust the unit tests, a reason they were never sufficient alone.

**Not manufacturing evidence stayed a real discipline, not just a stated principle.** Twice today
(declining to spin up a second entity to test the domain-boundary hypothesis; explicitly not
proposing a Decision Point mechanism on a "too early to tell" verdict) the cheaper, tempting move
was to act on incomplete evidence anyway. Both times the actual move was to name the gap and stop.
