## Context

Today was split between two threads that turned out to feed each other: continuing to drive
`canopy implement product-001` against a live dogfooding project (Node/Express
backend + React/Vite frontend), and building out Canopy's own console UX from scratch. The UX
work kept surfacing real skill/prompt bugs as a side effect of watching the tool run more
closely than a plain scrolling log ever made possible — several of today's most useful fixes
were found *because* the new checklist made a contradiction or an omission visually obvious,
not because anyone went looking for it.

## Shipped

### Prompt/skill fixes, found from live evidence
- **`modifiedAt` contradicting the story's own acceptance criteria** — the model-layer skill's
  worked example hardcoded `modifiedAt: null` at construction; this story's BDD scenarios say
  the system sets it to the current timestamp. The skill was silently overriding the story.
- **`categories` boundary-condition conflation** — one schema field with two independent
  validations (`max_length` per item, `max_items` for the array); the test-generation prompt
  didn't disambiguate them, so the model tested one constraint's boundary but asserted the
  other's error message.
- **`exactOptionalPropertyTypes` — the real root cause** — traced all the way back to the
  generated `tsconfig.json`, which had two strict flags (`exactOptionalPropertyTypes`,
  `noUncheckedIndexedAccess`) that nothing in the skill ever asked for. The model added them
  on its own initiative, then spent the rest of the day failing to satisfy the very strictness
  it introduced. Fixed the config-layer skill to explicitly forbid adding to the specified
  compiler-option set, with the reasoning stated inline.
- **The conditional-spread "fix" that didn't actually fix anything** — a plausible-looking
  attempt (`{ ...parsedProduct, ...(parsedProduct.description !== undefined && {...}) }`)
  that spread the *original* object first, which already carried the problem field — a
  conditional spread evaluating false can't remove a key an earlier spread already added.
  Subtler than "forgot the pattern," and worth its own WRONG/CORRECT pair in the shared rule.
- **Double-mounted routes, taught by our own skill** — the route layer's worked example showed
  `router.post('/widgets', ...)` for a router mounted at `app.use('/widgets', router)`,
  producing an actual path of `/widgets/widgets`. The skill was teaching the bug it later had
  to be fixed for.
- **Self-contradicting fix-loop output** — the model's self-reported summary was printed even
  when the byte-comparison (`is_noop`) already proved the file was untouched, producing lines
  like "model made no changes" immediately followed by "Fixed the TypeScript error by...".
  Self-report is only shown now when the file actually changed.
- **Missing typing pattern for mocking a class vs. an interface** — the route-test worked
  example taught mocking only `EventPublisher` (a plain object mock), never showing how to
  `jest.mock()` a whole *class* and type the result as `jest.Mocked<T>` — which is what the
  model actually needed once it (correctly) started mocking the service layer directly.

### A small step toward "rules as shared, not duplicated"
Consolidated the `exactOptionalPropertyTypes` rule — previously restated three separate times
with three separate WRONG/CORRECT examples — into one shared constant referenced by every
call site that needs it (model layer, route layer, test-gen prompt), plus extracted it into
`typescript_common_rules()` so React/Vite and Angular skills get it too, closing a gap where
they simply didn't have the rule at all. Explicitly *didn't* build the fuller generalized
`Rule` type (axis-tagged across language × tech × layer × phase) — one demonstrated axis
(language) isn't enough evidence to justify that scope yet; agreed on concrete triggers to
watch for before revisiting.

### Console UX, built from nothing
- Elapsed-time timers, TTY detection with a plain-line fallback, color/dim helpers via `console`.
- A live, persistent step checklist (`indicatif::MultiProgress`) replacing plain scrolling
  output — the whole plan visible from the start, the active step expanding with a spinner and
  phase text, a finished step collapsing to one line.
- Fixed a redraw bug where finished steps kept reappearing after every subsequent line —
  root cause: indicatif only permanently folds a bar into scrollback when its *last* strong
  handle is dropped, and every bar was being held for the run's whole lifetime.
- Viewport-aware lazy reveal: bars are created on demand within a window sized to the actual
  terminal height, with a trailing "N more step(s)" summary, so a huge plan can't push the
  active step off-screen.
- Per-step sub-operation history (test-gen, stub-gen, each fix attempt) that accumulates
  visibly under the active step and collapses away as a group on success, but is *kept* on
  failure as diagnostic context.
- Fix-attempt lines enriched with the actual compiler/test error text, not just a bare
  "fixing X" — the error was already being extracted, just never shown.
- Grouped checklist: steps nest under a permanent header naming their owning service and
  marking it backend vs. frontend at a glance — and the final cross-service regression pass
  now reuses the exact same mechanism instead of falling back to plain prints, closing an
  asymmetry between how Red/Green phases and the final check were presented.
- Found and fixed a real data-loss-on-display bug: `ProgressFinish::default()` is `AndClear` —
  any bar dropped without being explicitly finished vanishes with no trace. Every pending
  preview, group header, and the tail summary were silently disappearing the moment a broken
  build caused an early return. Added an explicit freeze step before returning.
- Fixed a duplicate-elapsed-time cosmetic bug (a bar's own template already appends the timer;
  a second, differently-formatted one was being embedded in the message on top of it).

### JS/TS tests moved to co-location, diverging from Java on purpose
Explicit user call: TypeScript should follow its *own* ecosystem's convention (test file next
to its source, e.g. `src/services/WidgetService.test.ts`), not Java's mirrored `src/test/java/`
tree — the two are allowed to disagree. Touched 7 files across `canopy-cli`/`canopy-llm`
(`tdd.rs`, `execute.rs`, `fix.rs`, `plan.rs`, `scaffold.rs`, `step.rs`, `testing.rs`,
`tech_stack.rs`); simplified `tsconfig.json` scaffolding to a single `include: ["src/**/*"]`
now that a separate `tests/**/*` entry is unnecessary. Java's own convention was left untouched.

### A route file that compiled but was structurally wrong
`products.ts` compiled clean (`tsc` has no opinion on export shape) but broke at runtime:
`TypeError: argument handler must be a function`. Root cause — the rule "route files must
default-export a `Router` instance, never a factory" lived only in the `app`-layer section of
the tech-stack skill, but layer-scoped rendering only sends a file its *own* layer's rules, so
a route-layer generation call never saw it. The model produced a
`registerProductRoutes(router)` factory, matching exactly the ✗ WRONG example the skill already
had — just filed under the wrong layer. Moved the rule to the `route` section where it's
actually read; trimmed the `app`-layer copy to a one-line cross-reference.

### Full reset when the ground shifted under an in-flight story
Once the co-location convention changed, `product-001`'s already-`done` steps (and its
`plan.yaml`, which had two now-obsolete explicit test steps for `admin-portal` baked in from
before the fix) were inconsistent with everything the fixed code would generate going forward.
Chose a full reset — delete all 5 generated files under `services/product/`, delete
`plan.yaml` so it regenerates from scratch — over hand-migrating 5 files, since the project's
whole purpose is validating Canopy itself and the file count was small enough that regeneration
cost less than the risk of a manual import-path mistake.

### Live time + token accounting per step
Added `LlmClient::token_totals()` (atomic counters, incremented on every API response's
`usage` field) so the console checklist could show aggregated input/output tokens and elapsed
time per step, not just per sub-operation. First pass only showed the aggregate on the
step's *collapsed* line; user caught that the *active* step's own header never updated live.
Fixed by switching the active step's bar to a ticking-elapsed style (`reset_elapsed()` +
`{elapsed_precise}` in the template) only once it's actually activated — not from the moment
it merely scrolled into the preview window — and refreshing the header's token count after
every sub-operation, not just on phase change.

### A second orphaned-println bug, same root cause as an earlier one
A bare `progress.println("✓ product")` fired every time a compile/test check passed on the
first try — printed via `MultiProgress::println`, which always renders *above* the whole
pinned checklist region, disconnected from whatever step it was actually about. Same bug class
`annotate_last_child` was built to fix earlier today for the model's self-report notes, just
never applied to this call site. Deleted it outright — the checklist's own collapse line
already reports success with more detail.

### Trimmed two real prompt duplications, found by reading the actual sent prompt
User's observation ("context window looks substantial per step") led to pulling the actual
`llm-debug.log` entry for the simplest possible file (`Product.ts`, one interface + one
factory, no dependencies) — 5,645 input tokens for a first LLM call. Found two concrete,
non-speculative duplications inside that one prompt:
- `EXACT_OPTIONAL_PROPERTY_RULE` (~230 words) sent twice — once via the tech-stack skill's
  `model`/`route` layer section, again via the test-gen prompt's own IMPORTANT bullet list.
  For those two layers, the bullet now just points back to the copy already given.
- Two near-duplicate worked examples for the same layer — `testing.rs`'s layer-scoped
  `node_express_layer_examples()` already provides a complete `model` example, but
  `step.rs`'s own `test_structure` builder had a hand-written `model` skeleton checked
  *before* the "defer to the existing example" branch, so both got sent. Reordered the
  `if`/`else` chain so `layer_has_worked_example` is checked first — the hand-written
  `model`/`event` skeletons now only fire for tech stacks without a dedicated example.

## Learned

**Two failure classes look similar but need different fixes.** Roughly half of today's real
bugs were "the model doesn't know this obscure TypeScript rule" (`exactOptionalPropertyTypes`,
Zod's `.optional()` widening). The other half were "the skill itself contradicts the story, or
teaches the bug directly in its own example" — a knowledge gap looks the same as an
authoring gap from the outside (both show up as a compile error), but only one of them is
fixed by writing a better rule. The other is fixed by fixing what's already there.

**Watching more closely finds more bugs than looking harder.** The self-contradicting
fix-loop output, the vanishing pending steps, and the duplicate timestamp were all found by
building better observability, not by auditing the code for correctness. A UX investment paid
for itself in bug discovery before it paid for itself in readability.

**Symmetry is a good diagnostic question.** "Is the TDD approach information-symmetric across
Red, Green, and final regression?" turned out to have a concrete, checkable answer (no — the
final pass was falling back to `Progress::none()` while Red/Green got the full treatment), and
fixing the asymmetry was small once named. Worth asking that question about other three-phase
or N-way things in this system.

**Even our own prompt-ordering fixes can reintroduce the bug they fixed.** The
"promise-then-interrupt" pattern (a colon promising an example, three unrelated sentences
before the example actually arrives) that we found and fixed twice earlier in this project
recurred a third time — self-inflicted, during the very refactor meant to generalize the rule
and reduce duplication. Consolidating text is not free of the ordering risk that motivated the
original fixes.

**Don't trust a library's documented-sounding default.** `ProgressFinish::default() ==
AndClear` is not what "just drop it" intuitively suggests. Verifying against indicatif's
actual source (not just its docs) turned "why did everything disappear" from a guess into a
one-line fix with a real citation.

**Generalize when you have two independent data points, not one.** The "should Rule be a
first-class citizen" question got a concrete answer by naming the specific triggers that
would justify the bigger investment (a second language hitting the same cross-tech
duplication; the phase axis duplicating a second time) rather than building the general system
from a single example.

**A layer-scoped rule is only as good as its layer assignment.** Two separate bugs today
(the route-file export-shape rule filed under `app` instead of `route`; the model-layer test
example duplicated because a hand-written fallback wasn't checking "does a real example
already exist" first) were really the same mistake: the *content* was correct, but it either
lived under the wrong key or wasn't given priority over a redundant fallback. Layer-scoping
only pays off if every call site actually reads from the layer it's scoped to.

**A UI number is only trustworthy once it updates where the user is actually looking.**
Showing tokens/time only on a step's collapsed line technically had the right data, but it
was invisible for the entire time anyone would want to watch it — while the step was active.
"Add the aggregate" and "add the *live* aggregate" are different asks even when the
underlying accumulator code is identical.

**Real per-call token counts turn "context feels big" into a fixable diagnosis.** Without
`token_totals()`, "the context window is substantial" would have stayed a vague impression.
With actual numbers per step and the ability to pull the exact prompt that produced them from
`llm-debug.log`, a vague concern became two specific, verified, low-risk deletions in one pass.
