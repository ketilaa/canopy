## Shipped

### Finished the Scenario Coverage Enumeration split, live-verified
Yesterday's half-done split (schema/policy generation in one call, a mechanically-computed coverage
matrix, then scenario generation in a second call) went through prompt review, three rounds of
fixes (a dropped anti-hallucination guardrail on the non-creation-story fallback path, leftover
domain-specific example vocabulary from an earlier prompt version, an inaccurate claim about which
system-generated fields are actually null at creation), and a live run against the real local
model. Confirmed working exactly as designed: a 2-mandatory/2-optional schema plus 3 resolved
policies produced a 12-item coverage matrix, and the model returned exactly 12 scenarios, one per
item, correctly grounded.

### Found and fixed a real bug in the project's own YAML repair utility
The live-verification run above crashed on the very case it was meant to prove: the model wrote a
genuinely empty scenario constraint list (`constraints: []`), and a pre-existing repair function —
meant only to quote bracket-style type annotations like `type: [string]` — quoted the empty list
into a broken string (`constraints: "[]"`) instead, because its "is this a real sequence or a type
annotation" check happened to pass vacuously on empty content. Fixed by excluding the empty case
explicitly; added 4 regression tests (this function had none before), including the exact case that
crashed.

### Forced per-item policy classification, then found it wasn't enough
Business Policy Discovery — the six-question business-rule checklist yesterday's retrospective
flagged as the next enumeration candidate — got the same fix Stage 0 already proved: instead of a
freeform resolved/unresolved split, the model must now output exactly six named entries, one per
fixed area, each with its own classification. That closed one bug (the model had been dumping half
its answers into an unlisted fourth bucket). It did not close a second, more serious one, caught by
inspecting a real generated result directly: the model was confidently "resolving" business
questions — a specific authorization role, a specific default value, a specific retention policy —
with zero support anywhere in the story, the ADRs, or the domain vocabulary it had been shown.

### Required evidence for every confident policy answer
Rebuilt the classification rule: "resolved" (and, after a second pass found the same hole from a
different angle, "not applicable" too) now requires naming its exact source — the specific
sentence in the story, ADR, or domain vocabulary that actually states the rule — and code checking
the response fails the whole operation if that citation is missing, rather than accepting an
ungrounded answer. A confirmatory reproducibility sweep showed this actually changed behavior, not
just the error message: before the fix, most runs "resolved" 5 of 6 questions with fabricated
specifics; after, runs resolved 1–2 of 6, with the rest correctly landed as open questions, and the
new citation check never once fired an error across the post-fix runs — meaning the model was
genuinely complying, not being blocked.

### Fixed the duplicate domain-event-ADR bug, across four review rounds
Yesterday's continuity work left one confirmed, quantified bug unaddressed: re-running `spec`
against a story that already had a domain-event ADR still produced a duplicate roughly 2 of 3 times,
even with the existing ADR shown verbatim in context. Replaced the holistic "check the ADR list
yourself" instruction with a mechanically-computed fact, injected once as a single "Domain Event
Status" line the model just has to act on. Getting the mechanical check itself correct took four
independent review passes, each catching a real gap the previous one missed: matching on entity
alone (which could wrongly suppress a legitimately different, later event for the same entity),
a case-sensitive comparison (which would silently reintroduce the bug against a hand-edited domain
registry), a raw-substring verb check (misclassifying "update ... address" as a creation, because
"add" is a substring of "address"), and a duplicated restatement of a naming rule already stated
elsewhere in the same prompt. A confirmatory sweep afterward: 0 duplicate ADRs across 3 runs, down
from the previously measured ~2/3 incidence.

### Refined the project's own example-noun convention
Looking back at yesterday's anchoring fix (a "safe" generic example noun still got echoed
verbatim), applied the same scrutiny to the project's existing convention of using a deliberately
fake placeholder noun ("Widget") in prompt examples — and found the identical risk in miniature: an
independent review flagged a prompt using "Widget" three times in one example set as recreating the
same anchoring pattern, just with a conventionally-safe word instead of a domain-specific one. House
style now prefers bracketed structural placeholders (`<entity>`, `<domain object>`) over any
concrete noun, even "Widget," reserving real words for cases where the structure itself — not the
subject matter — genuinely needs one.

### A confirmatory sweep tying the day's fixes together
Three runs from an identical fixed baseline, checking specifically whether the day's two fixes
held under real, repeated model sampling: zero duplicate ADRs in all three runs, and every policy
question landed in a sanctioned bucket with no fabrication-shaped output — a sharp, measured
improvement over the same sweep run before either fix existed.

### Turned this project's own history into external-facing drafts
A different kind of task: reviewed commit history, retrospectives, and reproducibility data to
identify which findings were genuinely reusable, evidence-backed engineering lessons rather than
routine progress. Produced three sample blog-post drafts (the enumeration story, the example-noun
story, the policy-fabrication story) and six graded engineering-principle documents, each required
to state real counter-evidence or an honest boundary case rather than defaulting to "none found."
Rewrote the blog drafts a second time after direct style feedback — lead with the surprising
concrete observation, not the conclusion; direct voice; evidence before principle. Then reorganized
all of it, plus the daily retrospectives themselves, into this `docs/` structure inside the actual
project repository, rather than leaving it in a separate personal notes folder.

## Learned

**An audit that requires evidence is only as strong as what counts as evidence.** Requiring a
citation for every confident policy answer sharply cut outright fabrication. It didn't fully close
the gap — some citations, inspected closely, quote real source text that doesn't actually support
the specific claim being made. A real improvement, not a complete fix; worth remembering that a
"require a citation" fix and a "verify the citation supports the claim" fix are two different levels
of rigor, and today only implemented the first.

**Getting a mechanical check right is itself an iterative process, not a one-shot win.** The
domain-event-ADR fact-computation function needed four independent review rounds before it was
correct — an operation-blind match, a case-sensitivity gap, a substring false positive, and a
duplicated rule statement, each found only after the previous fix was already believed done. Moving
a judgment from "ask the model" to "compute it in code" doesn't make the computation automatically
correct; it just moves the correctness burden somewhere code review and tests can actually reach it.

**The same anchoring mechanism recurs at different scales, and fixing the first instance doesn't
immunize the second.** Yesterday's fix ("register an account" as an anchor) and today's finding
("Widget" repeated three times as an anchor) are the same underlying failure, just triggered by
different words with different degrees of domain-specificity. Neither a domain-specific noun nor a
deliberately generic one is safe on its own — the fix that actually holds is removing the noun
entirely in favor of a structural placeholder.

**Turning engineering work into external-facing writing surfaces gaps the work itself didn't.**
Writing the engineering-principles documents, with a required counter-evidence section, forced
questions ("does this principle have a real boundary case, not just a hypothetical one?") that
hadn't been asked while the fixes themselves were being made. The enumeration principle's own
counter-example — enumerating inputs doesn't constrain output shape — came directly out of writing
the principle down carefully, not out of the original bug-fixing session.

  a cited source actually supports the specific claim made. Flagged as the clearest next test in
  both the policy-discovery blog draft and its corresponding open question, not yet started.
- **The blog-draft and principles-library pass was explicitly a sample, not the full taxonomy.**
  A complete pass identifying every candidate article and principle across the project's full
  history is still pending.
- **Contract-driven implementation is next, per explicit direction.** Wiring `canopy implement` to
  actually read `contracts.yaml` instead of the current ADR/architecture-skill-driven planning is
  the next major body of work, with a specific litmus test proposed for when it lands: if switching
  the target language or framework changes the contract itself, planning concerns have leaked back
  into implementation, and that's a sign the layer boundary isn't holding.
- Whether the "example noun as candidate answer" finding generalizes across model families and
  sizes, or is specific to smaller local models with weaker instruction-following, is untested —
  worth a controlled ablation if this becomes a recurring concern on a different model.
